# prd-first-methodology

Before any code gets written, Claude creates a PRD (Product Requirements Document) as a markdown file in the workspace. The PRD describes what is being built, the target user, existing repos being pulled from and what to take, stack recommendations, features to cut, phases from MVP to full, and ordered steps. Claude reads the PRD at the start of every session. The PRD does three jobs: forces decisions before code (easier to edit a document than refactor built code), creates persistent context Claude can re-read across sessions, and can be audited by a second Claude instance asking "what is overcomplicated, what is missing, what would you simplify?". Markdown is the format because it is low token cost, Claude reads it well, it is editable anywhere, and it persists.

## Claims

- PRD is a markdown file describing what is being built, target user, repos to pull from, stack, features to cut, phases, and steps [sources/stack-1-1-starting-the-build-process] [sources/stack-1-2-how-i-use-claude-code-in-build]
- Three jobs: forces decisions before code, creates persistent context across sessions, allows auditing by a second Claude [sources/stack-1-2-how-i-use-claude-code-in-build]
- Lives inside the workspace, read at the start of every session [sources/stack-1-1-starting-the-build-process] [sources/stack-1-3-designing-for-your-use-case]
- Turns a stateless tool into one with memory; the memory lives in the filesystem, not Claude's context window [sources/stack-1-3-designing-for-your-use-case]
- Markdown is chosen because it is small (low token cost), Claude reads it well, editable in any text editor, and convertible to formatted Google Docs [sources/stack-1-1-starting-the-build-process] [sources/stack-1-3-designing-for-your-use-case]
- When Claude drifts during a build, pointing back to the PRD re-aligns it [sources/stack-1-3-designing-for-your-use-case]
- A second Claude session can audit the PRD before code exists [sources/stack-1-2-how-i-use-claude-code-in-build]
- In the website-build walkthrough, the PRD is generated by the UX/UI Design Skill and includes design tokens, color values, typography, layout patterns, and a delivery checklist [sources/playbooks-3-1-build-and-deploy-website]
- Step 5 of the pre-build planning sequence requests the PRD and explicitly says "do not start building yet" to hold the checkpoint before code [sources/playbooks-3-3-pre-build-planning-and-prompt-sequencing]
- Editing the PRD is free; editing after code is expensive. Van Clief calls it the "last free checkpoint" [sources/playbooks-3-1-build-and-deploy-website] [sources/playbooks-3-3-pre-build-planning-and-prompt-sequencing]
