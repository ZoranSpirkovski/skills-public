# playbooks-3-1-build-and-deploy-website

> Source: Implementation Playbooks course, Module 3 Lesson 3.1. Written walkthrough (video key-moments listed). File: `.wiki/raw/playbooks/lesson-3-1-build-and-deploy-website.md`.

## Summary

The canonical end-to-end website build. Ten steps from zero to a live GitHub Pages URL using fewer than ten prompts. The move that compresses the run: do analysis and briefing in Claude chat (the browser product), hand off a markdown briefing file to Claude Code, and front-load scope, deployment target, and folder structure in the first prompt. A PRD generated by the UX/UI Design Skill acts as the last free checkpoint before code. The lesson flags every moment where "I should have done this earlier" appears as a token cost and treats the folder structure plus built-in project memory as the continuity layer across sessions.

## Key claims

1. Build starts in Claude chat, not Claude Code. The browser session explores the reference site and writes a markdown briefing.
2. The markdown briefing is how context transfers between Claude chat and Claude Code. Without it, the user reruns discovery in the Code session.
3. The first Claude Code prompt does four things at once: requests a folder structure with `CLAUDE.md`, states deployment target (GitHub Pages), sets scope boundaries, and ends with "ask me three questions."
4. The "ask me three questions" suffix forces Claude to read everything, process it, and surface gaps before building anything.
5. Before code, Claude generates a PRD using the UX/UI Design Skill: design tokens, color values, typography, layout patterns, delivery checklist. Editing the PRD is free; editing code is expensive.
6. Claude Code writes project memory to a local file automatically. No separate memory system is required; folder structure and `CLAUDE.md` handle routing, and built-in memory handles session continuity.
7. The first build will look wrong. Correct it with screenshots plus natural-language requests; the second iteration lands closer.
8. Every "I should have done this earlier" moment is a token cost. Planning saves money.
9. A simpler GitHub Pages route exists: create a repo, upload an `index.html`, enable Pages. Three minutes. Worth knowing even for CLI users because it clarifies what Pages does.
10. Claude Code installs the GitHub CLI, authenticates, creates the repo, sets up `.gitignore`, and pushes; subsequent deploys are one prompt.
11. CSS pathing issues on first deploy are common: local and GitHub environments have different URL prefixes; a screenshot plus a short message is enough for Claude to correct paths.
12. For a simple one-page site, skip Claude Code entirely and upload a Claude-chat-generated HTML artifact directly to GitHub Pages.
13. The total prompt count for a full project using this sequence is usually under ten.
